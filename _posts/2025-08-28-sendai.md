## Executive Summary

Sendai is a medium-difficulty Windows Active Directory machine on HackTheBox. The attack chain progresses through six distinct stages, each exploiting a realistic misconfiguration commonly found in enterprise environments.

The foothold begins with **anonymous SMB access** to a company share — a misconfiguration that exposes two files, `incident.txt` and `guidelines.txt`, which document a recent internal security incident in which user accounts with weak passwords were identified and expired. This contextual information is operationally significant: expired accounts in Active Directory enter a `STATUS_PASSWORD_MUST_CHANGE` state that allows password reset without knowledge of the current credential. With a valid guest session, **RID brute-force** enumerates all domain principals. A subsequent **empty-password spray** across the enumerated user list identifies two accounts — `Elliot.Yates` and `Thomas.Powell` — in the exploitable must-change state. The password for `thomas.powell` is reset via `impacket-changepasswd` using a null session, providing the initial authenticated foothold.

**BloodHound** analysis of the domain ACL graph reveals a privilege chain reachable from `thomas.powell`: the `SUPPORT` group (of which `thomas.powell` is a member) holds `GenericAll` over the `ADMSVC` group, and `ADMSVC` holds `ReadGMSAPassword` over the `mgtsvc$` Group Managed Service Account. `thomas.powell` is first added to `ADMSVC` via `bloodyAD`, then the GMSA NT hash for `mgtsvc$` is retrieved over LDAP. Since `mgtsvc$` is a member of the `REMOTE MANAGEMENT USERS` group, the hash is used directly for a **Pass-the-Hash WinRM** session on the Domain Controller.

Local enumeration as `mgtsvc$` targets the Windows Registry service tree. A single service entry at `HKLM\SYSTEM\CurrentControlSet\Services` contains **hardcoded credentials** in its `ImagePath` value — the executable `C:\Windows\helpdesk.exe` is invoked with the username `clifford.davey` and a plaintext password embedded as command-line arguments. These credentials are valid for the domain.

**BloodHound** now shows that `clifford.davey` is a member of the `CA-OPERATORS` group, which holds `GenericAll` over the `SENDAICOMPUTER` certificate template in the Active Directory Certificate Services (ADCS) deployment. `GenericAll` over a certificate template is the definition of **ESC4** — the attacker can modify the template's flags and permissions to introduce an ESC1 condition (enable enrollee-supplied subject, add `Client Authentication` EKU). After the template is modified using Certipy, a certificate is requested with the Administrator account's UPN and SID embedded as the subject. Authenticating with this certificate via PKINIT retrieves the Administrator's NT hash, which is used with **Pass-the-Hash** via Evil-WinRM to complete the domain compromise.

**Machine Details:** Windows Server 2022 (Build 20348), domain `sendai.vl`, Domain Controller hostname `DC` / `dc.sendai.vl`. Difficulty rated Medium.

## Understanding the Key Techniques

### What is STATUS_PASSWORD_MUST_CHANGE?

When an Active Directory administrator marks an account with "User must change password at next logon", the account enters a state tracked internally by the NTSTATUS code `0xC0000224`. In this state, Windows enforces a special authentication flow: the credentials are technically valid, but the domain controller rejects normal interactive logons and instead forces the user through a password change dialog. The critical security implication is that Windows allows the password to be changed via the DCE/RPC `NetUserSetInfo` call using a **null session** — no knowledge of the current password is required. Any tool that can speak the password-change RPC (such as `impacket-changepasswd`) can set a new password for any account in this state. The design rationale is to allow users who have forgotten their password to still complete the reset; the attack consequence is that identifying such accounts from an anonymous session and resetting their passwords is trivial.

### What is a GMSA and Why is ReadGMSAPassword Dangerous?

A **Group Managed Service Account (GMSA)** is a special Active Directory account type designed for services that must run across multiple hosts. The domain controller automatically manages the GMSA's password — a 240-byte random blob — rotating it on a configurable schedule (default 30 days). The password is stored in the `msDS-ManagedPassword` attribute of the GMSA object in Active Directory. The `msDS-GroupMSAMembership` attribute defines a security descriptor that specifies exactly which principals are authorised to read the managed password blob via LDAP. A principal with the **ReadGMSAPassword** edge in BloodHound (corresponding to being listed in `msDS-GroupMSAMembership`) can retrieve the managed password BLOB over LDAP and derive the NT hash, enabling Pass-the-Hash or Kerberos authentication as the GMSA account.

### What is ADCS ESC4 to ESC1?

**Active Directory Certificate Services (ADCS)** is a Microsoft PKI role that issues X.509 certificates from configurable **certificate templates**. Templates define which users can enroll, what key usages are allowed, how the subject name is constructed, and whether extended key usages include `Client Authentication`.

**ESC4** describes the condition where an attacker holds write permissions (typically `GenericAll`, `GenericWrite`, or `WriteProperty`) over a certificate template object in Active Directory. This allows the attacker to modify the template's `msPKI-Certificate-Name-Flag` and `pKIExtendedKeyUsage` attributes directly.

**ESC1** describes a template that, in its current state, allows an enrollee to supply an arbitrary Subject Alternative Name (SAN) — including any User Principal Name (UPN) — while also granting `Client Authentication` as an Extended Key Usage. A certificate issued with a target user's UPN in the SAN can be used in a PKINIT Kerberos authentication exchange to obtain a TGT as that user.

The **ESC4 to ESC1** attack chain: using write permissions from ESC4, the attacker modifies the template to introduce ESC1 conditions. They then enroll a certificate specifying the Administrator's UPN in the SAN. The CA issues the certificate because the template now permits it. The certificate is used with Certipy's `auth` module to perform PKINIT, obtaining a TGT and then, via U2U (User-to-User) Kerberos, extracting the NT hash of the Administrator — enabling Pass-the-Hash without ever knowing the plaintext password.

The combination of open ports immediately identifies this as a Windows Active Directory Domain Controller:

- **Port 53 (DNS):** Active Directory depends on DNS for service location records (SRV records). Every DC runs a DNS server.
- **Port 80/443 (HTTP/HTTPS, Microsoft IIS 10.0):** IIS serves the default Windows Server landing page. The HTTPS certificate CN is `dc.sendai.vl`, disclosing the fully-qualified DC hostname and the domain name `sendai.vl`.
- **Port 88 (Kerberos):** The Key Distribution Centre — definitive confirmation that this is a Domain Controller and the primary authentication protocol in use.
- **Port 389/636 (LDAP / LDAPS):** Active Directory LDAP interfaces for querying domain objects. The LDAP banner discloses the domain `sendai.vl` and the site `Default-First-Site-Name`.
- **Port 445 (SMB):** The file-sharing and named-pipe endpoint. This is the primary target for initial anonymous enumeration.
- **Port 464 (kpasswd5):** The Kerberos password change service. Its presence on every DC is standard.
- **Ports 3268/3269 (Global Catalog / GC over SSL):** Global Catalog ports allow domain-wide LDAP searches across all objects in the forest, not just the local domain.
- **Port 3389 (RDP):** Remote Desktop enabled on the DC. The NTLM challenge response in the RDP handshake discloses the domain name `SENDAI`, the NetBIOS name `DC`, and the OS build `10.0.20348` — confirming **Windows Server 2022**.
- **Port 5985 (WinRM):** Windows Remote Management — the PowerShell remoting endpoint. If credentials are obtained for an account in the `REMOTE MANAGEMENT USERS` group, Evil-WinRM provides an interactive shell here.
- **Port 9389 (.NET Message Framing):** The Active Directory Web Services endpoint, used by PowerShell AD cmdlets and tools like PowerView.

The RDP `rdp-ntlm-info` output provides full domain identification without any credentials: `DNS_Domain_Name: sendai.vl`, `DNS_Computer_Name: dc.sendai.vl`, OS `Windows Server 2022 Build 20348`. SMB signing is enforced — NTLM relay attacks against this host are not viable.

## Initial Foothold

### RID Brute-Force for Domain User Enumeration

To mount a targeted password spray, a complete list of domain usernames is needed. Since no credentials are available yet, RID brute-force is used over the guest SMB session.

Every Active Directory security principal — user, group, computer account — is identified by a **Security Identifier (SID)** of the form `S-1-5-21-<domain>-<RID>`. The final numeric component is the **Relative Identifier (RID)**, which is unique within the domain. Well-known RIDs (500–520 for built-in principals) are fixed; domain accounts start at 1000 and increment sequentially. By iterating RIDs and querying the DC for the associated SID via the `LsaLookupSids` RPC, all domain objects can be enumerated — even as a guest:

## Privilege Escalation — ADCS ESC4 to ESC1 to Domain Administrator

### BloodHound Analysis for clifford.davey

Querying BloodHound for `clifford.davey`'s outbound edges reveals:

`clifford.davey` is a member of `CA-OPERATORS`, which holds `GenericAll` over the `SENDAICOMPUTER` certificate template.

`GenericAll` over a certificate template object is the definition of **ESC4**: the attacker can write any attribute of the template, including those that control enrollment permissions and subject name handling. Modifying these attributes to introduce ESC1 conditions — specifically enabling the `CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT` flag and ensuring `Client Authentication` is listed as an Extended Key Usage — creates a template from which a certificate with an arbitrary UPN can be requested.

### Understanding the ESC4 to ESC1 Certificate Forgery

Before executing the attack, the `SENDAICOMPUTER` template is queried to confirm the current state and identify the Certificate Authority name:

```bash
kali@kali$ certipy-ad find -u 'clifford.davey' -p 'RFmoB2WplgE_3p' -dc-ip 10.129.234.66 -vulnerable
```

The CA name `sendai-DC-CA` and the template name `SENDAICOMPUTER` are confirmed.

**Step 1 — Save the existing template configuration and modify it to enable ESC1.** Certipy's `template` subcommand with no explicit modification flags automatically backs up the existing configuration and applies the standard ESC4 to ESC1 modifications: it sets `msPKI-Certificate-Name-Flag` to include `CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT` (value `0x1`), sets `msPKI-Enrollment-Flag` to remove `CT_FLAG_INCLUDE_SYMMETRIC_ALGORITHMS` if present, adds the `Client Authentication` OID (`1.3.6.1.5.5.7.3.2`) to `pKIExtendedKeyUsage`, and grants enrollment rights to `Domain Users`:

```bash
kali@kali$ certipy-ad template -u 'clifford.davey' -p 'RFmoB2WplgE_3p' -dc-ip 10.129.234.66 -template SENDAICOMPUTER -save-old
Certipy v4.8.2 - by Oliver Lyak (ly4k)

[*] Saved old configuration for 'SENDAICOMPUTER' to 'SendaiComputer.json'
[*] Updating certificate template 'SendaiComputer'
[*] Successfully updated 'SendaiComputer'
```

The original configuration is saved to `SendaiComputer.json` so it can be restored after the certificate is obtained, minimising the window of misconfiguration.

**Step 2 — Request a certificate with the Administrator's UPN and SID.** With the template now permitting enrollee-supplied subject names, a certificate enrollment request is sent to the CA specifying `administrator` as the UPN and the Administrator's object SID as the certificate SID. The SID is included to satisfy Windows Kerberos's SID-based principal mapping introduced as a hardening measure in newer patch levels:

## Vulnerability Chain Summary

**Step 1 — Anonymous SMB Exposes Internal Documents:** Guest read access to the `sendai` share allowed retrieval of `incident.txt`, which disclosed that a batch of accounts had been expired due to weak passwords — directly informing the next phase of the attack.

**Step 2 — RID Brute-Force via Guest Session:** The domain's null/guest SMB session permitted `LsaLookupSids` RPC calls, enabling enumeration of all domain users, groups, and service accounts without any credentials.

**Step 3 — STATUS_PASSWORD_MUST_CHANGE Exploitation:** Two accounts — `Elliot.Yates` and `Thomas.Powell` — had the "must change password" flag set following the security incident. The Windows DCE/RPC password change protocol permits resetting such passwords via a null session, providing authenticated domain access for `thomas.powell`.

**Step 4 — GenericAll to ReadGMSAPassword to GMSA Pass-the-Hash:** The `SUPPORT` group (containing `thomas.powell`) held `GenericAll` over `ADMSVC`, which in turn held `ReadGMSAPassword` over `mgtsvc$`. Adding `thomas.powell` to `ADMSVC` via LDAP, then reading the GMSA managed password blob, yielded the NT hash for `mgtsvc$` — usable directly for WinRM Pass-the-Hash authentication.

**Step 5 — Hardcoded Credentials in Service ImagePath:** A custom service `helpdesk.exe` was registered with the domain credentials for `clifford.davey` embedded as plaintext command-line arguments in the registry `ImagePath` value, readable by any authenticated user on the system.

**Step 6 — ESC4 to ESC1 ADCS Certificate Forgery, Domain Admin:** The `CA-OPERATORS` group (containing `clifford.davey`) held `GenericAll` over the `SENDAICOMPUTER` certificate template. This ESC4 condition was exploited to modify the template, enabling ESC1 (enrollee-supplied subject). A certificate was enrolled for the Administrator account and used via PKINIT to extract the Administrator NT hash, completing the domain compromise via Pass-the-Hash.

## Mitigations

**Disable anonymous and guest access to SMB shares.** The `sendai` and `Users` shares permitted guest read access, exposing internal documents and enabling RID brute-force enumeration. All non-default SMB shares must require authentication. Guest access should be disabled at both the share-level ACL and the domain Group Policy (`Network access: Restrict anonymous access to Named Pipes and Shares`).

**Prevent null-session password resets by enforcing authentication on the DCE/RPC password change interface.** The `STATUS_PASSWORD_MUST_CHANGE` state should not permit null-session resets. Where possible, enforce that password changes require the current credential or a verified out-of-band reset token, even for expired accounts. Monitor for `NetUserSetInfo` RPC calls originating from null sessions.

**Audit and disable anonymous RID enumeration.** The `LsaLookupSids` and `NetrGetDCName` interfaces that power RID brute-force should not be reachable by unauthenticated or guest sessions. The `Network access: Restrict clients allowed to make remote calls to SAM` policy restricts this interface to specific security groups.

**Enforce least-privilege group membership and audit GenericAll/WriteProperty ACEs on privileged groups.** The privilege chain from `SUPPORT` holding `GenericAll` over `ADMSVC`, which holds `ReadGMSAPassword` over `mgtsvc$`, represents a multi-hop escalation path that should not exist. Each ACE grant should be reviewed: `GenericAll` over a group that controls a GMSA is an indirect path to the GMSA's credentials. BloodHound should be run regularly to identify and eliminate such privilege chains.

**Never store credentials in service ImagePath registry values.** Credentials embedded in service startup command lines are visible to any user who can query `HKLM\SYSTEM\CurrentControlSet\Services`. Services requiring domain account credentials should use GMSAs (which have no stored password) or retrieve credentials from a secrets manager. The existing `mgtsvc$` GMSA approach is architecturally correct for this purpose.

**Audit and restrict write permissions on ADCS certificate templates.** `GenericAll` over a certificate template is equivalent to the ability to forge certificates for any account in the domain. Write access to certificate templates must be restricted to Domain Admins and the Enterprise PKI Administrators group. The `Cert Publishers` group should be reviewed. Run `certipy find -vulnerable` or Locksmith regularly to detect vulnerable template configurations before attackers do.
